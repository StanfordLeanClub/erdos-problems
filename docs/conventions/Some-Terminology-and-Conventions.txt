- a "hit" / "attack" = an attempt to push the frontier forward by identifying and thoroughly investigating the current most promising approach, which may or may not be followed by a report of what was found and what the next best attack is. 

- a "combo" = multiple hits in a row; instead of reporting back after a hit, whatever would have been reported as the next best attack is immediately followed up on as the current most promising approach. e.g., 2-hit combo is: think hard on current best attack -> prove stuff -> now there's a next best attack -> think hard on current best attack -> prove stuff -> report back, a 3-hit combo is: think hard on current best attack -> prove stuff -> now there's a next best attack -> think hard on current best attack -> prove stuff -> now there's a next best attack -> think hard on current best attack -> prove stuff -> report back, etc. 

- an "n-combo" = an n-hit combo; 2-combo = 2-hit combo, 3-combo = 3-hit combo, etc. 

- CONVENTION: the depth and quality of each hit should be prioritized. i.e., don't worry about overall combo time; in general, I'd rather have full combos take longer with deep and meaningful hits than quick combos with shallow hits.

- a "step-back" / "step back" (taking a step back) = considering all that's been done thus far and seeing 1) if the current results can be consolidated or generalized, 2) if there are more promising threads for the current approaches for the current, and 3) if there's a more promising approach that those currently in consideration. This is done to re-center us and keep the big picture of what we ultimately need to show in mind so that we (hopefully) avoid proving a bunch of lemmas in an increasingly narrow direction that might be a dead end. 

- a "frontier note" / latest "state of the frontier document" = document written in latex (or markdown) including everything proven thus far + path of discovery + citations, made such that a fresh and skeptical context of you (or anything sufficiently capable reasoner) can start from the statement of the relevant Erdos problem and/or its webpage on https://www.erdosproblems.com/ and follow the document's arguments all the way to where we are currently. Additionally, you should also produce a frontier visualizer update YAML file which summarizes the latest frontier notes info in a way that can be used for the visualization dashboard.
